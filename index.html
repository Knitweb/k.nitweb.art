Agentic workflow

The workflow is part of the art.

Knitweb development should look like a protocol studio: terminal-first work, memory files, skills, isolated worktrees, visual evidence, adversarial review, and public PR proof. The repositories are not separate chores; they are coordinated agents in one build loop.

Open workflow knitweb.dev route 100 professions
  1. Issue to worktree to PR
  2. Tests plus visual evidence
  3. Protocol contracts before UI claims
  4. Public progress via docs and news

Serverless dapp model

One state model, many clients.

Protocol first

Identity, signatures, proofs, wallet derivation, and useful-work records belong to Knitweb/Pulse. UI shells are consumers, not authorities.

Transport replaceable

A hosted relay can help bootstrap peers, but the dapp must not make one host the permanent trust root.

Native where useful

Android and desktop clients can be thin native shells around the same signed state, not separate backends with divergent truth.

Why Nit is here

A stable outsider reference.

Nit is not the protocol. It is a useful language reference because its calculator example separates business logic from desktop, Android, and iOS presentation. That is the same client architecture we want for Knitweb dapps. The repository graph makes those clients point back to the real protocol repositories instead of becoming disconnected demos.

  • Shared core logic
  • Platform refinements
  • Android APK target
  • No server-owned account state

Build path

From reference to proof.

  1. Positioning site Publish the model and reference language story.
  2. Protocol fixture Define a small signed-state JSON fixture any client can render.
  3. Android shell Build a native or Nit-inspired shell around the same fixture.
  4. P2P transport Replace hosted defaults with explicit peer/relay selection.
